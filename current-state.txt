Model Weights (/models): The training outputs are fully serialized to disk:
gnn_drug_repurposing.pt (~29 MB): Contains the trained PyTorch state dict and model configuration (like num_nodes, hidden_dim, etc.).
adjacency.pt (~74 MB): A pre-calculated sparse tensor representing the normalized graph edges.
metadata.pkl (~1.5 MB): Mappings of human-readable text to internal IDs, plus node definitions (e.g., mapping "Anemia" to the correct node index).

 The Backend API (Actively Running)
The backend is built with FastAPI (backend/main.py) and is currently running on your system via uvicorn main:app --reload --host 0.0.0.0 --port 8000.

Initialization: On application startup, it loads the PyTorch checkpoints, sets up the PrimeKGDrugRepurposingGNN architecture, and evaluates the node_type_ids alongside the adjacency matrix to get the pre-calculated multi-dimensional encode representation for the nodes.
Static Assets: It serves the /data directory statically, so consumers can download raw .pdb (proteins) and .sdf (ligands) files.
The /predict Endpoint (POST):
It accepts a JSON body with a disease string and a top_k integer.
It performs a smart lookup on the disease_nodes (exact match, followed by substring match).
It constructs pairwise interactions for the target disease against all drug_nodes, mapping them into a tensor, and passes them to the model's torch.sigmoid(model.score()) link predictor.
It maps disease queries to known AutoDock targets (e.g., Cancer -> TP53: 1TUP).


Note on Validation: While the README implies real-time AutoDock Vina processing, 

"the FastAPI backend currently simulates ("mocks") the binding affinity calculations (by generating randomized scores normalized to -7.5 ± 1.5 kcal/mol baseline) yielding quick predictive results without physically blocking API requests on CPU-intensive heavy molecular physics simulations."

4. Neural Network Architecture Details
The runtime uses a PyTorch architecture composed of:

Node & Type Embeddings: Dedicated nn.Embedding combinations initializing node identity features.
Two-Layer GCN: Two sparse graph convolutions wrapping intermediate node representations.
Link Predictor (MLP): Concatenates [drug_node, disease_node, drug_node_Hadamard_product] to output a single scalar value.

The Graph Neural Network (GNN) can score 10,000+ drugs in a fraction of a second. However, AutoDock Vina performs heavy 3D molecular physics simulations. Running just one drug against one protein target can take anywhere from 2 to 15 minutes using 100% of a CPU core.

If the FastAPI endpoint legitimately ran AutoDock Vina for the top 10 candidates before responding, your API request would hang for over an hour and the browser would time out and crash.

How to make it accurate?
To integrate the true physics-based calculations into the API without breaking it, you have three architectural options:

Option 1: The Asynchronous Worker Approach (Recommended for Production)
Instead of running the chemistry physics in the main web thread, we use a message queue:

Step 1: You hit /predict with disease=cancer.
Step 2: The GNN instantly returns the Top 10 drugs without docking scores and adds a "Docking Job" to a background queue (e.g., using Celery/Redis or FastAPI BackgroundTasks).
Step 3: A separate background worker picks up the job, downloads the 3D structures, and runs AutoDock Vina for the next hour.
Step 4: Your frontend polls a new endpoint like /status/{job_id} until the genuine physics calculations are stored in a database and ready to display.
Option 2: Pre-computation (Database Lookup)
If you only have a set list of diseases and drugs, you can write a script that runs AutoDock Vina offline locally over the weekend for the top 1,000 combinations. When the FastAPI endpoint is hit, it simply queries a SQLite database for the pre-calculated true affinity.

Option 3: Synchronous Blocking (Only for Local Testing)
If you are only running this locally and are willing to wait 10+ minutes for an API response, we can directly inject the run_autodock_vina() logic from the existing .py training scripts directly into the FastAPI endpoint, replacing the random number generator block.
